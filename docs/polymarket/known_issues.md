# Known Issues — 已知資料品質與系統限制

> 此文件記錄已發現但尚未（或無法）修正的問題。其目的不是讓問題消失，而是讓使用資料的人**知道資料的限制在哪**。
>
> **重要原則**：我們不重寫歷史。當發現 bug 時，記錄問題的時段 + 影響欄位 + 處理建議，而非靜默修改既有資料。
>
> 對應 docs/polymarket/architecture.md 第一章原則 6（數據不可變性）。

## 紀錄格式

每個 issue 包含：
- **發現日期**：何時被發現
- **影響時段**：什麼時候開始有問題、什麼時候修好（若已修）
- **影響欄位**：哪些 SQL 欄位 / 計算結果不可信
- **影響範圍**：受影響的 wallet / market / scanner_version
- **處理建議**：使用該段資料時的注意事項
- **狀態**：active（仍存在）/ fixed-going-forward（修好但歷史資料仍有問題）/ resolved（已重算或無歷史殘留）

## 系統性限制（永久）

這些不是 bug，是系統設計的固有限制。記錄在此提醒未來分析者。

### 倖存者偏差
我們只能看到「仍在 Polymarket 交易且帳戶尚有資金」的錢包。已爆倉退出的錢包對未來預測有教訓意義，但我們的資料看不到。所有「歷史勝率」都隱含這個偏差。

### 代理錢包混淆
Polymarket 用代理錢包系統，一個真實用戶可能控制多個錢包地址。我們看到的是代理錢包行為，不是真實用戶完整行為。如果用戶在多錢包間切換，我們會把同一行為主體誤判為多個獨立錢包。

### 市場結構漂移
Polymarket 本身在演化（流動性、參與者、手續費）。一個錢包在 2024 年的行為模式可能在 2026 年已不適用。所有跨年度比較都應加註此免責。

### 解算延遲與爭議
UMA Oracle 解算少數市場會有爭議，期間倉位凍結 PnL 無法計算。當前系統把這類市場視為「未結算」，可能短期低估有效樣本數。

### scanner_version 邊界
scanner_version 升級時新舊計算邏輯不同。比較同錢包跨版本的 profile 必須先確認版本一致；混合不同版本的彙整統計沒有意義。
